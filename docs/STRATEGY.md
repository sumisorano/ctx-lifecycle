# 功成身退 · 实战策略

> 什么时候该做什么——让 `/ctx-check` 和 `/ctx-clean` 成为习惯，而不是事后补救。

---

## 为什么需要策略？

功成身退的命令本身很简单：`/ctx-check` 诊断，`/ctx-clean` 回收。

但**什么时候跑、跑多频繁、什么情况下用什么方案**——这些才是省 token 的关键。本文件回答"什么时候该做什么"。

---

## 分层策略总览

| 层 | 名称 | 触发条件 | 操作 |
|----|------|---------|------|
| 0 | **预防** | 写 Skill 时 | 精简体积 + 声明生命周期 |
| 1 | **检测** | 加载 ≥ 3 个 Skill | `/ctx-check` |
| 2 | **回收** | ctx-check 给出建议 | `/ctx-clean` |
| 3 | **对话管理** | 几十轮后仍慢 | 新会话 + 上下文摘要 |
| 4 | **环境优化** | 墙内 / 代理场景 | 调整 Claude Code 模式 |

---

## 第 0 层：预防（写 Skill 时就注意）

Skill 规则写得越胖，后面就越难瘦。

### 规则精简

```yaml
# ❌ 胖写法：示例 + 长篇解释全塞正文
lifecycle:  # ✅ 瘦写法：核心规则为主
  type: one-shot
  completion_signal: "评审报告已输出"
  retention_summary: "按 {skill} 完成了一次 {任务}"
```

- **核心规则**放正文
- **示例 / 长篇解释**放 `references/`
- **一句话能说清的**别写三段

### 声明生命周期

在 SKILL.md 的 frontmatter 中声明：

```yaml
lifecycle:
  type: one-shot                    # one-shot / session-long
  completion_signal: "..."          # 什么标志表示任务完成了
  retention_summary: "..."          # 回收后保留的一句话摘要
```

有声明 → `/ctx-check` 优先以声明为准，不需要 AI 推断，更准更快。

> 模板见 `lifecycle-template/SKILL.md`

## 第 1 层：主动检测（不等超时）

这是**最关键的习惯**。别等变慢了再查，提前发现可回收的 Skill。

### 检测时机

| Skill 加载数 | 操作 |
|-------------|------|
| 0~2 个 | 无需操作 |
| **3 个** | **主动跑 /ctx-check** ✅ |
| 4+ 个 | 每完成一个任务就跑一次 |

### 为什么是 3 个？

实测数据：

| Skill 数 | 对话轮次 | 响应时间 | 状态 |
|----------|---------|---------|------|
| 1~2 个 | 1~3 轮 | < 10s | ✅ 流畅 |
| 3 个 | 4 轮 | ~15s | 🟡 开始变慢 |
| 4~5 个 | 5~6 轮 | >60s（超时）| 🔴 卡顿 |

**3 个是拐点。** 此时回收已完成 Skill，能阻止后续卡顿。

### 两种门控模式

`/ctx-check` 内置轻量门控：

```
Skill 加载数 < 3 且 对话轮次 < 10
  → "🧘 上下文还很清爽"（跳过诊断）

Skill 加载数 ≥ 3
  → 进入完整诊断
```

> 注意：使用 `-c`（continue）模式时，实际对话轮次可能远多于显式用户消息数，按保守估计处理。

## 第 2 层：主动回收（有枣没枣打一杆）

### 回收原则

```
/ctx-check 说"建议回收" → 立即 /ctx-clean
```

不要犹豫，不要等。一次回收 3~4 个 Skill 大概释放 **18k~22k tokens**。

### 安全兜底

`/ctx-clean` 内置三层保护：

1. 🛡️ 当前活跃 Skill 绝不回收
2. 🛡️ 不确定是否完成的，宁可保留
3. 🛡️ 已回收的 Skill 留一行摘要，随时可重新加载

所以不存在"回收错了把还在用的也删了"的风险。

### 预期效果

| 指标 | 回收前 | 回收后 |
|------|--------|--------|
| 4 个 Skill 规则体积 | ~2,900 tokens | ~240 tokens（4 行摘要）|
| 响应速度 | >60s（超时）| <15s |
| 后续轮次可保持 | 越来越慢 | 稳定 |

## 第 3 层：对话管理（几十轮后的终极方案）

如果回收完仍然慢（比如对话已经三四十轮了，即使 Skill 回收了，历史对话本身也很大）：

```bash
# 让 Claude 总结进展，新开会话
claude -p "总结刚才的工作进展和关键决策，新建一个会话继续"
```

或者**精简 CLAUDE.md**：

```markdown
# 项目 CLAUDE.md

<!-- 把已经用完的 Skill 注释掉，只留还用得着的 -->
<!-- <skill name="code-review">... -->

<skill name="deploy-check">...</skill>
```

## 第 4 层：环境优化（墙内 / 代理场景）

### 问题

在墙内通过代理（如 cc-proxy 转 DeepSeek）使用 Claude Code 时：
- 代理本身增加延迟
- 上下文越大，代理转发 + 模型处理的双重延迟越明显
- pty 交互模式可能握手失败（ERR_BAD_REQUEST）

### 建议

| 模式 | 推荐度 | 说明 |
|------|--------|------|
| `claude -p "xxx"` | ⭐⭐⭐ | 单次查询，最快 |
| `claude -c -p "继续"` | ⭐⭐⭐ | 续接会话，推荐用于日常工作 |
| `claude`（pty 交互） | ⭐ | 墙内可能不可用 |

**推荐长期使用 `-c -p` 模式**，既能保留上下文，又不受 pty 握手问题影响。

## 完整工作流

```
开始工作
  │
  ├── 加载 Skill A → 完成任务
  ├── 加载 Skill B → 完成任务
  ├── 加载 Skill C → 完成任务
  │
  ├── ⚡ 已加载 3 个 Skill
  │     └── /ctx-check
  │           ├── 有已完成 → /ctx-clean → 继续
  │           └── 无已完成 → 继续
  │
  ├── 加载 Skill D → 完成
  │     └── /ctx-check → /ctx-clean
  │
  ├── ... 继续工作 ...
  │
  └── 几十轮后仍慢
        └── 新会话 + 摘要
```

**三个核心习惯：**

1. 🟢 **写 Skill 时就想好怎么走** — 瘦规则 + 声明生命周期
2. 🟡 **3 个 Skill 查一次** — 不等超时
3. 🟠 **有建议就回收** — 舍不得孩子套不着狼
